**产业链智能投研工作台**

可落地实现方案（MVP 至生产化）

版本：v1.0  
日期：2026-06-05

| **产品定位** | 面向投资研究的产业链理解、财务验证和事件传导工作台。                                  |
|--------------|---------------------------------------------------------------------------------------|
| **MVP 范围** | A 股 + 一个产业链 + 100–300 家公司 + 5 年财务 + 8 个季度 + 年报/公告/专家资料冷启动。 |
| **核心原则** | 证据优先、关系可追溯、模块权责清晰、模块间只通过结构化数据交互。                      |
| **不做事项** | 不做自动荐股、不输出买卖指令、不用 LLM 直接生成财务事实。                             |

# 1. 方案摘要

**目标：**建设一套“公司—产品—材料—产业链节点—事件—财务指标”联动的数据与应用系统，帮助用户回答公司做什么、处于什么产业链位置、关系是否有证据、产业链位置是否兑现为财务结果、事件如何沿产业链传导。

用户问题 → 图谱定位 → 证据追溯 → 财务验证 → 事件传导 → 投研解释

- 数据层负责采集和标准化，不负责投资解释。

- 抽取层负责产生候选实体/关系，不直接写入高置信图谱。

- 审核治理层负责把候选关系转为已验证关系。

- 图谱层负责关系查询和路径计算，不负责生成自然语言结论。

- 分析层负责把图谱、财务和事件组合成解释性结果。

- 应用层只通过 API 获取结构化结果，不直接访问底层原始文件。

# 2. 产品形态与使用方式

第一版产品建议做成三页一工作台：公司页、产业链页、事件页。后续再增加材料/产品页和个人关注页。

| **页面/入口** | **用户输入**             | **系统输出**                                                         | **核心价值**                                 |
|---------------|--------------------------|----------------------------------------------------------------------|----------------------------------------------|
| 公司页        | 公司名、股票代码         | 主营业务、核心产品、产业链位置、上下游、财务趋势、事件影响、证据来源 | 把“看不懂公司”变成“看得懂产业位置和财务兑现” |
| 产业链页      | 行业/产业链名称          | 上中下游图谱、关键环节、代表公司、瓶颈节点、风险路径                 | 让用户理解行业结构和价值分布                 |
| 事件页        | 新闻/公告/事件描述       | 事件分类、影响节点、传导路径、受益/受损公司、观察指标                | 把新闻从“信息”变成“影响路径”                 |
| 材料/产品页   | 材料、部件、设备、产品名 | 用途、下游场景、相关公司、受影响事件、关系置信度                     | 解决材料和零部件公司业务难理解的问题         |

# 3. 总体架构

系统采用“原始数据不丢、标准数据可查、候选关系可审、正式图谱可追溯、应用层不绕过接口”的架构。

数据源连接器  
→ Raw Store 原始数据层  
→ Parser & Chunker 文档解析层  
→ Extractor 候选抽取层  
→ Candidate Store 候选关系层  
→ Review & Governance 审核治理层  
→ Entity / Graph Store 实体与图谱层  
→ Event Engine 事件传导层  
→ Insight Engine 财务验证与解释层  
→ API Gateway  
→ 前端工作台  
  
财务 ETL → Financial Metrics Store → Insight Engine / Company Page

# 4. 模块权责划分

每个模块只对自己的结构化输出负责。模块之间不通过共享文本、临时文件或隐式状态耦合。所有关键数据都必须带 schema_version、trace_id、source_id、created_at。

# 7. 端到端流程设计

| **流程**       | **步骤**                                                                             | **产物**                                         | **质量门槛**                                           |
|----------------|--------------------------------------------------------------------------------------|--------------------------------------------------|--------------------------------------------------------|
| 财务数据流程   | 采集财务 API → raw 保存 → 字段映射 → 指标计算 → 异常检测 → 公司财务卡片              | financial_raw, financial_metrics, metric_anomaly | 核心指标缺失率 \< 5%；异常变动必须标记 quality_flag    |
| 文档到图谱流程 | 下载/上传文档 → 解析切片 → 实体抽取 → 关系抽取 → 实体归一化 → 审核 → 写入 graph_edge | document_chunk, candidate_relation, graph_edge   | 正式关系 100% 有 evidence_chunk_id；高置信关系需 ≥ 0.8 |
| 事件传导流程   | 采集事件 → 分类 → 映射影响节点 → 图谱路径传播 → 生成影响路径 → 财务指标匹配          | market_event, event_impact_path, company_insight | 每条影响路径必须能展开到边和证据                       |
| 用户查询流程   | 用户输入 → API 网关 → 图谱/财务/事件查询 → Insight Engine 组合 → 前端展示            | API response JSON                                | 前端不得拼接业务逻辑；所有结果可追溯                   |

# 8. 关系建立规则

关系是本系统的核心资产。所有关系必须经过“候选 → 审核 → 正式图谱 → 过期/降权”的生命周期。

pending 候选  
→ approved 已审核通过  
→ verified 正式图谱  
→ stale 长期未出现，自动降权  
→ expired 过期  
  
rejected 被拒绝  
needs_more_evidence 证据不足，等待补充

| **证据类型**            | **基础加分** | **说明**                                         |
|-------------------------|--------------|--------------------------------------------------|
| 年报/招股书明确披露     | +0.40        | 最高优先级，用于产品、主营、客户、供应商、风险。 |
| 公告/重大合同明确披露   | +0.30        | 适合订单、扩产、客户变化、合作关系。             |
| 公司官网/投资者关系记录 | +0.20        | 适合产品、应用场景、技术方向。                   |
| 表格规则抽取            | +0.20        | 如前五大客户、供应商、财务表。                   |
| 专家资料/课程/研报摘要  | +0.10~0.20   | 适合行业本体和候选关系冷启动。                   |
| 新闻/媒体报道           | +0.05~0.15   | 需注意时效和可信度。                             |
| 只有 LLM 推断           | 上限 0.40    | 不得直接进入高置信图谱。                         |
| 近 2 年重复出现         | +0.10        | 说明关系仍然有效。                               |
| 3 年以上未出现          | -0.20        | 自动降权或转入 stale。                           |

# 9. API 与模块接口

前端、外部应用和分析引擎统一通过 API 网关访问。建议先做 REST，后续再增加 GraphQL 或 Cypher 查询代理。

| **接口**                     | **方法** | **输入**                    | **输出**                           | **用途**       |

# 13. 质量、合规与风控

- 证据强制：candidate_relation 和 graph_edge 必须有 evidence_chunk_id 或 source_record_id。

- 事实与观点分离：年报/公告为事实源；专家材料和研报为观点/候选源。

- LLM 输出不直接入高置信图谱：必须经过置信度规则和审核。

- 财务数字不得由 LLM 生成：只能来自结构化接口、官方披露或确定性计算。

- 投资建议边界：输出“产业链位置、财务质量、事件敏感度、风险提示”，不输出买卖指令。

- 数据授权：生产环境需要确认财务、新闻、研报、价格等数据源授权范围。

- 可追溯：每个前端结论都可以展开到指标、关系边、证据片段和来源文件。

# 16. 主要风险与应对

| **风险**             | **表现**                         | **应对**                                                 |
|----------------------|----------------------------------|----------------------------------------------------------|
| 关系质量不足         | 图谱看起来丰富但很多边不可信     | 证据强制、置信度分层、人工审核、低置信关系不进入正式图谱 |
| 数据授权风险         | 商业化无法使用爬取数据或研报内容 | MVP 与生产数据源分离；尽早确认授权；保留官方公告证据     |
| 本体过细导致维护困难 | 行业节点无限膨胀                 | 先做人可维护的行业本体；每个新类型必须有业务用途         |
| LLM 幻觉             | 把行业常识误写成公司事实         | LLM 只输出候选；证据片段必须原文支持；高置信需多源验证   |
| 产品变成复杂大屏     | 用户看图但解决不了问题           | 围绕公司、事件、行业三个问题设计；图谱只是解释路径       |
| 合规边界模糊         | 用户理解为荐股                   | 避免买卖建议；输出风险、因子、证据和不确定性             |
