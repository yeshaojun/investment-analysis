## 1. 技术栈规范

- 框架：`Next.js`（`App Router`）
- 状态管理：`Zustand`
- 语言：`TypeScript`（严格模式）
- UI：
  - `Tailwind CSS`
  - `Radix UI`
  - `shadcn/ui`
- 测试：
  - 单元 / 组件测试：`Jest` + `React Testing Library`
  - 端到端测试：`Playwright`

## 2. 项目定位与设计基线

- 前端必须按`复杂生产项目`设计，不按 demo、单页展示站、一次性交互原型设计
- 默认服务对象是持续使用的真实用户，而不是只跑 happy path 的内部演示
- 页面设计和代码实现必须考虑：
  - 高频访问
  - 多页面跳转
  - 长会话交互
  - 异步加载与失败重试
  - 权限差异
  - 首屏性能
  - 可观测性
- 所有页面都要同时考虑：
  - 桌面端
  - 移动端
  - 弱网场景
  - 长列表 / 大数据量场景

## 3. 写法规范

- React 组件必须使用函数式 + Hooks 写法
- Next.js 必须遵守 `Server Component / Client Component` 规范
  - 默认：`Server Component`
  - 只有以下情况允许使用 `"use client"`：
    - 交互事件（click、input、drag、selection）
    - `Zustand`
    - 浏览器 API
    - 必须在客户端执行的第三方 UI 能力
- 自定义 Hooks 必须以 `use` 开头
- 禁止在条件、循环、嵌套函数中调用 Hooks
- 组件、Hook、工具函数、类型定义职责必须分离，禁止把页面、状态、请求、映射逻辑揉进一个文件
- 新增代码默认补充类型，不允许为了省事使用 `any`

## 4. 目录与分层规范

- 前端代码必须分层，至少区分：
  - `app/`：路由、布局、页面入口
  - `src/components/ui/`：原子 UI 组件
  - `src/components/base/`：基于 UI 组件的通用封装
  - 各业务目录下 `_components/`：业务组件
  - `src/lib/`：纯工具函数、格式化、常量
  - `src/hooks/`：通用 Hook
  - `src/stores/`：全局客户端状态
  - `src/types/`：共享类型
- UI 组件、base 组件必须先沉淀到 `src/components`
- business 组件存放在各自业务目录下，其他模块复用前先判断是否应上提
- 禁止建立“万能 utils 文件”或“万能 hooks 文件”
- 页面层不应直接处理复杂数据转换，映射逻辑应抽离到独立函数或 Hook

## 5. 组件封装规范

- 组件必须进行分层设计：`ui`、`base`、`business`
- 组件要合理拆分，逻辑适度抽离，保持小而专注
- 超过 `180` 行 JSX 的组件必须拆分
- 复杂页面默认采用容器组件（Container）与展示组件（Presentational）分离
- 组件对外暴露的 props 必须语义清晰，禁止传递大而混乱的配置对象
- 组件必须优先可组合，而不是通过大量布尔参数堆出分支
- 禁止在组件内部硬编码接口地址、业务文案映射、权限判断规则
- 业务组件应明确三种状态：
  - loading
  - empty
  - error

## 6. 页面与路由规范

- 路由层只负责页面装配，不负责承载复杂业务逻辑
- `layout` 只放跨页面共享结构，不在其中塞入页面专属状态
- 页面必须按用户任务组织，不按接口返回结构组织
- 复杂页面应拆成：
  - 页面入口
  - 业务区块
  - 数据映射层
  - 交互控制层
- 页面级错误必须可恢复，不能只显示空白页或报错栈
- 对需要 SEO / 首屏性能的页面，优先使用服务端渲染和服务端取数

## 7. 数据获取规范

- 服务端数据优先通过 `Server Component`、服务端函数或受控请求层获取
- `Zustand` 只用于客户端状态，不用于替代服务端数据真相源
- 禁止把以下内容塞进 `Zustand` 作为长期主数据源：
  - 商品列表真值
  - 订单详情真值
  - 权限真值
  - 服务状态真值
- 所有请求必须统一经过受控封装，禁止页面中散落 `fetch` 调用
- 请求封装至少统一处理：
  - 超时
  - 错误映射
  - 鉴权信息
  - trace 信息透传
  - 降级提示
- 前端不得假设后端永远成功，必须显式处理：
  - 慢响应
  - 空结果
  - 部分结果
  - 权限失败
  - 降级返回

## 8. Zustand 规范

- 必须使用 selector 方式获取状态，避免全量订阅
- store 按业务拆分，避免单一巨大 store
- store 中只存客户端共享状态，不存可以从 props、URL、服务端数据直接推导出的内容
- 禁止把异步请求、DOM 引用、一次性表单临时值全部堆进全局 store
- store 必须定义清晰的 state、actions、reset 策略
- 涉及登录态、会话态、筛选条件的 store，必须明确持久化边界与清理时机

## 9. 表单与交互规范

- 表单必须有完整的输入校验、错误提示、提交中状态和重复提交保护
- 交互反馈必须及时，不能让用户对系统当前状态产生猜测
- 危险操作必须有确认和可撤销策略
- 长耗时交互必须分阶段反馈：
  - 已接收
  - 处理中
  - 已完成 / 失败
- 禁止只展示“请求失败”，必须告诉用户下一步可以做什么
- 对 AI 导购、对话、推荐类页面，必须明确区分：
  - 系统在思考
  - 系统在检索
  - 系统在等待后端
  - 系统已降级

## 10. 用户体验规范

**用户体验是所有产品的最高准则，优先级高于技术偏好、代码整洁度和架构优雅度。后端可以很复杂，但用户触碰到的每一层必须丝滑。**

- 为目标设计，不为功能设计：先问“用户要完成什么”，再决定页面怎么做
- 不要让用户思考：交互应该不言自明，需要说明书才能使用就是失败
- 系统承担复杂性：能自动化的不手动，能推断的不让用户填，能一步完成的不拆成三步
- 渐进式展示：先给核心，细节按需展开，不一次性把所有选项甩给用户
- 反馈引导行动：不要只报错，要告诉用户当前发生了什么、接下来怎么办
- 对关键决策页面，优先降低认知负担，而不是展示更多信息

## 11. 页面 UI 规范

- 页面设计以及修改必须遵守 [DESIGN.md](./DESIGN.md)
- 不允许“管理后台默认风”“AI 通用组件风”直接上线
- 视觉层级必须清晰，重点信息、次级信息、辅助信息一眼可分
- 同一业务流中的按钮、表单、卡片、提示、空态必须保持一致语言和视觉规则
- 页面必须考虑以下视觉状态：
  - 默认态
  - hover / focus / active
  - disabled
  - loading
  - error
  - empty
- 对长列表、对比表、推荐卡片，必须优先保证可扫读性而不是花哨布局

## 12. 渲染性能规范

- 避免不必要的重复渲染
- 列表渲染必须使用稳定 `key`，禁止使用 `index` 作为 key
- 纯展示组件在确认收益后使用 `React.memo`
- 合理使用 `useMemo` / `useCallback` 做性能优化，但禁止机械滥用
- 避免在 render 中创建新对象、匿名函数和高成本计算
- 大列表必须考虑：
  - 分页
  - 分段加载
  - 虚拟滚动
  - 图片延迟加载
- 首屏只加载当前页面真正需要的资源，禁止把非首屏能力全部打进初始包
- 性能优化必须基于可观测问题，不做“玄学优化”

## 13. 可访问性规范

- 所有可点击元素必须可键盘访问
- 表单必须有明确 label、错误提示和焦点状态
- 图标按钮必须有可读语义
- 颜色不能作为唯一信息载体
- 弹窗、抽屉、下拉等浮层必须处理：
  - 焦点管理
  - ESC 关闭
  - 遮罩点击策略
  - 屏幕阅读器语义

## 14. 权限与安全规范

- 前端不得把权限控制只当作 UI 显隐
- 任何订单、售后、账户、管理能力都必须同时依赖后端权限校验
- 前端必须处理未登录、无权限、会话失效、接口拒绝等状态
- 禁止在前端硬编码敏感配置、密钥、内部接口地址
- 本地存储中禁止落敏感信息明文
- 对用户输入必须防御：
  - XSS 风险
  - 富文本注入
  - URL 参数污染
  - 非法跳转

## 15. 可观测性规范

- 前端关键交互必须有统一埋点策略，禁止页面作者各写各的事件名
- 至少跟踪以下关键行为：
  - 搜索
  - 筛选
  - 推荐点击
  - 对比操作
  - 加购 / 下单前动作
  - 错误重试
  - AI 对话关键分支
- 埋点必须区分：
  - 曝光
  - 点击
  - 提交
  - 成功
  - 失败
- 错误监控必须能定位到页面、组件、接口和用户动作上下文

## 16. 测试规范

- 组件必须包含单元测试
- 测试至少覆盖：
  - 渲染结果
  - 用户交互
  - 边界情况
  - 错误态
  - loading / empty 状态
- 核心页面必须补充集成测试或端到端测试
- 以下变更必须同步检查测试：
  - 路由改动
  - 组件公共 API 改动
  - 表单提交流程改动
  - 推荐 / 对比 / 对话主流程改动
  - 权限逻辑改动
- 不能只测 happy path

## 17. 发布与工程质量规范

- 所有页面在合并前至少通过：
  - 类型检查
  - lint
  - 单元测试
  - 关键链路人工验证
- 样式改动不能只看本机截图，必须确认：
  - 桌面端
  - 移动端
  - 空数据
  - 长文本
  - 错误态
- 引入新组件或新模式时，先确认是否与既有规范冲突
- 禁止带着临时 mock、调试按钮、未收口文案、伪 loading 状态进入主分支

# 全局 Claude 规则

## 1. 先想清楚用户流，再写页面

- 先明确用户目标、关键步骤、成功标准
- 不要拿到需求就先堆组件
- 对复杂页面先拆信息层级、状态流转和交互反馈

## 2. 简单优先，但不是简陋优先

- 只做需求范围内的功能
- 不为了“架构完整”提前造轮子
- 但不能为了省事省掉错误态、空态、权限态、loading 态

## 3. 精准修改

- 只改和需求相关的代码，不顺手重构无关页面
- 保持现有组件风格和目录边界
- 改完后清理无用 import、变量、样式和临时分支

## 4. 目标驱动

- 把需求转换成可验证目标，例如：
  - “优化列表体验” -> “明确首屏加载、空态、分页和错误重试”
  - “加一个对比功能” -> “补入口、状态、回退和埋点”
  - “改 AI 页面” -> “明确追问、等待、推荐、失败、降级几个状态”
