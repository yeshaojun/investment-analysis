# 后端技术规范

本规范面向 AIbuy 后端主链开发，默认服务对象不是 demo，而是需要稳定支撑 `1 万+` 用户、具备持续演进能力的生产系统。项目级业务原则、架构红线以 [README.md](/Users/andy/study/AIbuy/README.md) 为准；本文件定义后端实现层面的统一技术约束。

## 1. 技术栈规范
- 语言：`Python 3.11+`
- Web 框架：`FastAPI`
- 数据库：`PostgreSQL`
- 向量能力：`pgvector`
- 缓存：`Redis`
- ORM / 数据访问：统一使用一套受控数据访问层，禁止业务代码直连 SQL
- 后台任务：统一任务入口，异步任务与在线请求严格分离
- 配置管理：环境变量 + 受控配置模块，禁止硬编码生产配置
- 日志与监控：结构化日志 + 指标 + trace，默认接入统一观测链路

## 2. 规模目标与设计基线
- 系统必须按 `1 万+ 注册用户` 的复杂业务场景设计，不能按单机 demo 心态实现
- 默认按以下基线约束系统设计：
  - 日常存在持续并发访问，而不是单用户串行调用
  - 高峰期会出现对话、检索、推荐、订单查询混合流量
  - 任一单点依赖抖动都不能让主链全部不可用
- 核心接口必须定义并持续验证以下指标：
  - 可用性目标
  - `P50 / P95 / P99` 延迟
  - 错误率
  - 超时率
  - 降级触发率
  - 单请求成本
- 所有容量、超时、连接池、限流、缓存、任务并发配置必须显式化，禁止依赖框架默认值直接上线

## 3. 架构分层规范
- 首版架构采用`模块化单体`，先保证业务闭环、稳定性和边界清晰，不无故拆微服务
- 后端代码必须按职责分层：
  - `api`：路由、鉴权入口、请求响应模型、参数校验
  - `conversation`：意图识别、追问控制、工具编排、答案组织
  - `application services`：检索、排序、对比、订单、售后、搭配、购买等业务编排
  - `domain`：核心实体、领域规则、值对象、领域约束
  - `infra`：数据库、Redis、pgvector、外部 provider、adapter、任务基础设施
- 禁止以下逻辑混写：
  - 路由和业务编排
  - Prompt 和数据库访问
  - 排序规则和 HTTP 细节
  - mock 数据脚本和生产服务逻辑
- 对话层只能调用受控 service / tool，不能直接查库、拼 SQL、访问外部系统

## 4. 目录与模块边界规范
- 若新增模块，优先落到以下目录边界：
  - `src/api/`
  - `src/conversation/`
  - `src/shopping_guide/`
  - `src/retrieval/`
  - `src/ranking/`
  - `src/orders/`
  - `src/bundles/`
  - `src/domain/`
  - `src/infra/`
  - `src/tasks/`
- 模块之间只能通过明确接口通信，禁止跨目录直接调用对方私有实现
- 共享能力必须先判断是否真共享，禁止为了“看起来优雅”提前抽象公共层
- 单个模块过大时，优先按能力拆分，不按“utils 大杂烩”方式收纳

## 5. API 设计规范
- 对外 API 必须语义稳定、版本明确、错误码可枚举
- `FastAPI` 路由层只负责：
  - 请求校验
  - 鉴权鉴权上下文注入
  - 调用 application service
  - 响应序列化
- 禁止在路由函数中编写：
  - 多表查询
  - 排序决策
  - Prompt 拼装
  - Redis 业务逻辑
- 所有输入输出模型必须使用显式 schema，禁止返回松散字典作为正式契约
- 所有接口必须定义：
  - 超时预算
  - 幂等要求
  - 限流策略
  - 错误返回格式
  - 日志脱敏规则
- 写接口必须优先考虑幂等性；订单、服务预约、状态写入类操作必须具备幂等键或唯一约束

## 6. 编码规范
- 遵循 `PEP 8`
- 变量、函数、模块名使用 `snake_case`
- 类名使用 `PascalCase`
- 公共函数、类、模块使用 Google 风格 `docstring`
- 新增核心逻辑必须补充类型标注
- 禁止裸 `print()`，统一使用日志组件
- 禁止无边界 `except Exception` 吞错；捕获异常必须转换为可观测、可定位、可回传的错误
- 常量、阈值、缓存 TTL、路由开关必须集中管理，禁止散落硬编码

## 7. 数据建模规范
- 数据模型遵循“共享骨架 + 品类扩展字段”模式
- 至少围绕以下主实体建模：
  - `product_base`
  - `sku_inventory`
  - `order`
  - `user_decision_event`
  - `product_embedding`
- 商品事实、库存事实、服务事实、订单事实必须分层存储，禁止在同一字段混入多种语义
- 所有关键表必须定义：
  - 主键策略
  - 唯一约束
  - 更新时间字段
  - 来源字段
  - 软删 / 状态字段策略
- schema 设计必须考虑未来接入真实外部系统，禁止把 mock 字段直接当作长期契约
- 涉及推荐、排序、订单、售后的核心数据表，必须明确索引策略，不能等慢查询后再补

## 8. Repository / Provider / Adapter 规范
- 数据访问统一经过 `repository / provider / adapter`
- repository 负责受控读写，不负责业务编排
- provider 负责外部数据源或能力源接入，不向上暴露底层协议细节
- adapter 负责 mock、第三方系统、异构结构之间的转换
- 禁止 service 直接依赖第三方 SDK 细节
- mock 数据必须放在 provider / adapter 边界之后，禁止业务层 if/else 区分“mock 模式”

## 9. 事务与一致性规范
- 单请求内需要保证原子性的写操作必须显式事务化
- 跨数据库、缓存、向量索引、异步任务的更新，必须定义一致性策略：
  - 先写库再发任务，还是事件驱动更新
  - 失败补偿怎么做
  - 重试是否幂等
- 禁止把“先写 Redis、以后再补库”作为正式方案
- 进入交易前链路时，库存、价格、配送、安装能力必须二次校验，不能直接复用旧推荐结果
- 任何会影响用户最终决策的事实，都必须带时间戳或版本信息

## 10. 检索、排序与 RAG 规范
- 检索主链必须固定包含：
  - 关键词检索
  - 向量检索
  - 结构化过滤
  - 业务规则
  - 排序
- 排序顺序必须固定为：
  - 硬约束过滤
  - 需求匹配排序
  - 业务目标二级排序
- 向量召回只负责找候选，不得替代结构化过滤与事实校验
- Embedding 至少按以下粒度拆分：
  - `product_summary`
  - `product_features`
  - `scenario_fit`
  - `service_policy`
- RAG 只用于解释、政策问答、补充说明，不得把 RAG 文档当作订单状态或库存事实源
- 推荐结果必须可追溯到：
  - 请求上下文版本
  - 检索策略版本
  - 排序策略版本
  - 数据快照版本

## 11. 缓存规范
- 缓存是主链能力，不是“上线后再优化”
- 至少设计以下缓存层：
  - `query_parse_cache`
  - `retrieval_cache`
  - `recommendation_cache`
  - `rag_answer_cache`
- 缓存 key 必须基于标准化需求画像、过滤条件、策略版本和数据版本，不能直接依赖原始自然语言问题
- 缓存设计必须明确：
  - TTL
  - 主动失效条件
  - 版本切换策略
  - 热点保护策略
  - 穿透、击穿、雪崩防护策略
- 高风险事实禁止长时间缓存：
  - 库存
  - 价格
  - 配送时效
  - 安装可达性
  - 订单状态
- Redis 不可用时必须有降级路径，不能让接口无上限阻塞

## 12. 异步任务规范
- 以下操作默认走异步任务，不阻塞用户主请求：
  - Embedding 生成
  - RAG 文档切片和索引
  - 缓存预热
  - 数据导入
  - 回归评估
  - 批量同步外部系统数据
- 所有异步任务必须具备：
  - 唯一任务标识
  - 重试策略
  - 幂等保障
  - 死信或失败补偿机制
  - 运行日志和指标
- 禁止在请求线程中偷跑长耗时任务
- 任务结果若影响线上推荐质量，必须支持版本隔离和回滚

## 13. 性能与容量规范
- 性能优化优先级：
  - 先消除错误架构
  - 再减少 I/O 次数
  - 再优化 SQL / 索引
  - 最后才是局部微优化
- 所有数据库查询必须避免：
  - 全表扫描上线
  - N+1 查询
  - 无分页大结果集返回
  - 不受控排序字段
- 必须显式配置：
  - Web 并发 worker
  - 数据库连接池
  - Redis 连接池
  - 外部依赖超时
  - 线程池 / 协程并发上限
- 所有核心链路在上线前必须有压测结果，不允许“本地跑通”直接视为可上线
- 性能问题优先通过数据、索引、缓存、批处理、并发控制解决，不依赖无限扩容掩盖

## 14. 降级与容错规范
- 所有外部依赖都必须定义失败后的用户可感知结果，不能只返回“系统错误”
- 至少为以下依赖定义降级策略：
  - LLM
  - pgvector
  - Redis
  - RAG 文档源
  - 外部订单 / 服务 provider
- 降级策略必须明确：
  - 超时阈值
  - 熔断条件
  - 重试次数
  - 是否允许回退到较弱结果
  - 对用户的提示文案原则
- 当事实不可确认时，必须返回“未知 / 需进一步查询”，而不是猜测
- 降级不能突破业务红线，例如：
  - 跳过库存校验
  - 跳过权限校验
  - 跳过服务可达性校验

## 15. 安全、权限与隐私规范
- 所有订单、售后、安装、账户相关接口必须先鉴权再执行业务逻辑
- 必须区分：
  - 匿名会话权限
  - 登录用户权限
  - 管理后台权限
  - 内部任务权限
- 严禁 LLM 直接接触原始敏感数据
- 日志、trace、缓存、错误上报中必须脱敏：
  - 手机号
  - 地址
  - 订单号
  - 支付信息
  - 设备标识
- 所有外部输入默认不可信，必须防护：
  - Prompt 注入
  - 参数注入
  - 越权查询
  - 接口枚举
  - 恶意重放
- 密钥、数据库凭据、第三方令牌只能通过受控配置注入，禁止入库、入日志、入代码

## 16. 可观测性规范
- 每个请求必须具备统一 `request_id / trace_id`
- 核心链路必须记录结构化日志，至少包含：
  - 路由
  - 用户上下文类型
  - 意图
  - 耗时
  - 降级路径
  - 依赖调用结果
  - 策略版本
- 必须建立以下指标：
  - 请求量
  - 错误率
  - 超时率
  - 降级率
  - 缓存命中率
  - 数据库慢查询
  - 外部 provider 失败率
  - 单请求 token / 成本消耗
- 所有核心告警都必须可操作，禁止只报“接口异常”这种无法定位的问题

## 17. 测试规范
- 后端不是 demo，测试必须覆盖主链与失败路径
- 至少包含以下测试层次：
  - 单元测试：领域规则、排序逻辑、格式转换、工具函数
  - 集成测试：API + service + repository + Redis / DB 交互
  - 契约测试：外部 provider / adapter 边界
  - 回归测试：导购推荐、订单查询、RAG 问答、降级路径
  - 压测：核心读接口、写接口、对话主链
- 测试必须覆盖：
  - 正常路径
  - 边界输入
  - 超时 / 依赖失败
  - 权限错误
  - 幂等场景
  - 数据版本切换
- 检索、排序、Prompt、Embedding、RAG、缓存 TTL / key 策略变更时，必须同步补回归验证

## 18. 发布、配置与运维规范
- 所有配置必须区分环境：`local / test / staging / production`
- 禁止通过改代码切环境
- 功能开关、策略版本、灰度配置必须可控、可审计、可回滚
- 数据发布、索引切换、缓存预热必须形成标准流程，禁止人工随意操作线上主链
- 上线前必须完成：
  - 迁移脚本检查
  - 配置检查
  - 回归测试
  - 压测或容量评估
  - 回滚预案
- 任何影响推荐质量或服务事实的变更，都必须可追溯到责任人、版本和发布时间

## 19. AI 业务特殊约束
- 需求不明确时，最多追问 `2` 轮
- 明确需求后，默认只返回 `3-5` 个推荐
- 推荐理由必须绑定证据来源，不能做“常识性补全”
- 严禁编造以下事实：
  - 商品参数
  - 库存状态
  - 配送时效
  - 安装能力
  - 售后范围
  - 价格优惠
  - 订单状态
- 用户需求匹配度始终高于毛利、促销、库存清理等业务目标
- 不满足用户硬约束的商品，禁止进入最终推荐

## 20. 禁止事项
- 禁止把后端写成“单文件 + 全局变量 + 几个接口”的 demo 结构
- 禁止对话层、Prompt 层直接访问数据库
- 禁止在业务代码中散落 SQL、Redis key、Prompt 模板和第三方调用细节
- 禁止为了演示效果跳过库存、权限、服务可达性、交易前确认
- 禁止把一次性脚本、临时调试逻辑、实验代码混入主链
- 禁止没有监控、没有回归、没有回滚预案就发布核心链路

# 全局 Claude 规则

## 1. 先确认链路，再写实现

- 先判断需求影响的是 API、service、domain、infra 哪一层
- 先明确是否触发缓存、检索、排序、RAG、权限、异步任务影响面
- 高风险改动先补测试或回归约束，再改实现

## 2. 最小闭环，不做假闭环

- 优先做最小可上线闭环，不做伪架构
- 但“最小闭环”不等于跳过幂等、日志、鉴权、异常处理、测试
- 对复杂主链，宁可少做功能，也不能牺牲边界和可观测性

## 3. 精准修改

- 只改与需求相关的模块，不顺手重构无关部分
- 保持现有目录与分层风格
- 改完后清理无用 import、死代码、临时变量、调试残留

## 4. 目标驱动

- 把后端需求转换成可验证目标，例如：
  - “加缓存” -> “定义 key、TTL、失效策略，并补命中 / 降级测试”
  - “接订单接口” -> “补鉴权、幂等、超时、provider 契约和错误映射”
  - “优化推荐” -> “补回归样本、策略版本和效果对比”
